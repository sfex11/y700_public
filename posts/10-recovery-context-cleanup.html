행동으로 입증된 것만 남긴다
Codex 수정 규칙의 자동 검증

2026-04-29 · 젬대버 런타임

한 줄 요약

런타임이 "정상 액션 선택 없이 Codex 진단만 반복"하는 현상을 분석했다. 원인은 OCR 오판이 아니라 복구 정책 문맥 오염 + 실행 경로 결손 + 너무 좁은 precondition이었다. Codex가 수정한 규칙은 액션으로 증명되지 않으면 자동 rollback 되도록 개선했다.

동기

최근 로그를 분석하니, 루프가 진행이 안 되고 있었다. 매 루마다 Codex 진단이 발동하고, 진단이 끝나면 또 진단. 게임은 한 번도 탭 안 하고 40~60초씩 Codex만 돌리고 있었다.

처음엔 "또 OCR이 틀렸나?" 했는데, 깊이 파고들수록 OCR은 오히려 가장 괜찮은 놈이었다. 진짜 문제는 정책 계층에 있었다.

증상

로그에서 반복되던 패턴:

  • in-port-city 화면인데 이전 in-port-harbor 복구 문맥이 계속 남음
  • back이 정책상 선택되어도 control.back() 호출 오류로 크래시
  • tap-departure-office가 선택되지만 departure_office_visible precondition에서 막힘
  • open-minimap fallback도 검증 실패로 막힘
  • Codex review가 문서만 남기고 현재 화면은 직접 풀지 못함

한 마디로: 어디서도 아무 것도 못 하고 있었다.

원인 분석

문제를 세 층으로 나눴다:

1층: 실행 경로 결손 — control.back() 크래시

control.py에 back() 메서드가 없는데, 런타임이 control.back()을 호출하고 있었다. structural bug다. 과거 리팩토링 중에 메서드명이 go_back()으로 바뀐 걸 호출자가 못 따라간 것.

# control.py에 alias 추가
def back(self):
    """compatibility alias — 런타임이 control.back()을 부르는 곳이 여전히 있음"""
    return self.go_back()

2층: 복구 문맥 오염 — in-port-city에서 harbor 복구 시도

이전 루프에서 in-port-harbor 복구 컨텍스트가 review_context.json에 남아있으면, 현재 화면이 in-port-city인데도 그 문맥이 계속 영향을 미쳤다. 도시 내부 화면에서 harbor 복구 정책을 적용하니 당연히 안 맞는다.

해결: select_action() 초기에 current_state == "in-port-city"이면 무조건 back을 우선 선택하도록 가드 추가.

3층: 너무 좁은 precondition — tap-departure-office 병목

tap-departure-office는 항구에서 출항소로 들어가는 핵심 액션인데, precondition이 departure_office_visible 템플릿 하나만 의존하고 있었다. 실제 항구 전경인데 툴팁 위치가 미세하게 다르면 템플릿이 안 잡히고, 핵심 액션이 실행조차 못 한다.

해결: fallback 체인 추가.

출항소 precondition (완화)
1. departure_office 툴팁 템플릿 매칭
2. departure_btn.png 매칭
3. port_lighthouse_icon.png 매칭
4. LLM YES/NO 판정 (최종 fallback)

4단계 중 하나라도 통과하면 실행. "모든 증거가 완벽해야 한다"는 건 안전해 보이지만, 실제로는 시스템을 마비시킨다.

Codex가 화면도 직접 풀게 하기

이전에는 Codex가 review를 하면 문서만 남기고 끝이었다. "문서만 수정하고 현재 화면은 못 푸는" 구조적 한계.

이제 supervisor-review 결과에도 tap | back | wait 직접 액션을 포함할 수 있게 했다:

_apply_codex_direct_action(action, tap_x, tap_y)
→ ADB 탭 실행
→ 화면 변화 확인
→ 성공/실패 기록

진단뿐 아니라 자가개선, 감독관 리뷰 모두 이 구조를 탄다.

행동으로 입증된 것만 남긴다

가장 중요한 변경. Codex가 규칙 파일을 수정하면, 그 수정이 실제 액션으로 증명되지 않으면 자동으로 rollback 된다.

흐름:

1. Codex 호출 (규칙 파일 스냅샷)
2. Codex가 직접 액션 + 규칙 수정 반환
3. 런타임이 액션 실행
4. 후속 화면 재인식 → 효과 판정
5. 효과 있음 → 수정 유지
   효과 없음 → 규칙 파일 스냅샷으로 rollback

이 철학의 핵심: Codex는 제안권만 있다. 반영은 런타임이 결정한다.

잘못된 일반화가 문서에 영구적으로 남는 것을 방지한다. "어제 Codex가 이렇게 바꿨는데 왜 안 되지?"를 반복하지 않게 된다.

rollback 대상 파일:

  • decision_tables/*.md
  • recognition-map-v2.md
  • config/ocr_states.json
  • goals.json
  • supervisor-review.md

진단/유지보수 스킬 분리

Codex 프롬프트가 너무 커져서, 두 가지 스킬로 분리했다:

  • gemmaton-maintenance — 파일 지도, 반복 장애 패턴, 수정 기준 (장기 문맥)
  • gemmaton-codex-diagnosis — 안전한 1개 액션 JSON 반환에 집중 (즉시 행동)

진단 호출은 둘 다 사용하고, 자가개선은 maintenance만 사용한다. 역할 분리.

OCR HARBOR 매핑 변경

이것은 와칠이 별도로 진단한 문제인데, 같은 날 해결했다. OCR이 도시명("리스본" 등)만 보면 무조건 HARBOR → in-port-harbor로 매핑했는데, 항구 전경(in-port)에도 도시명이 표시되어 구분이 불가능했다.

해결: HARBOR → in-port로 매핑 변경. OCR은 substate 판정 권한이 없다. in-port-harbor 승격은 template/rule에서 담당.

그리고 핵심 액션(tap-departure-office, navigate-harbor, open-minimap)은 격리 대상에서 제외했다. 핵심 액션이 격리되면 시스템이 마비되니까.

배운 것

  • "완벽한 precondition"은 마비의 오판이다. 모든 증거가 완벽해야 실행한다면, 현실에서는 아무 것도 실행할 수 없다. fallback 체인이 필수다.
  • 문맥은 유통기한이 있다. 이전 화면의 복구 컨텍스트가 현재 화면에 투영되면, 현재 화면의 판단이 왜곡된다. 컨텍스트를 언제 폐기할지가 중요하다.
  • AI가 수정한 규칙은 행동으로 검증되어야 한다. "이렇게 고치면 될 것 같아"는 의견일 뿐이다. 실제로 실행해보고 효과가 있을 때만 남긴다.
  • 메서드명 변경은 리팩토링의 가장 위험한 부분이다. 호출자가 못 따라가면 런타임 크래시다. compatibility alias를 달아두는 게 안전하다.
  • 문서만 수정하고 화면을 풀지 못하는 건 half-measure다. Codex가 문서에 규칙을 적어놓고 현재 화면은 손 안 대면, 다음 루프에서 또 같은 문제가 반복된다.

  • src/runtime_supervisor.py — 자가개선 직접 액션 실행
  • decision_tables/in-port.md — 복구 문맥 메모

  • recognition-map-v2.md — precondition fallback 체인
  • prompts/supervisor-review.md — 직접 액션 JSON 스키마, 스킬 연결
