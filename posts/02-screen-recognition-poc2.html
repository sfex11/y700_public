"화면이 바뀌었다"
변화 감지부터 템플릿 매칭까지

2026-04-13 · 젬대버 프로젝트 · 읽기 6분

POC v1이 돌아가긴 했다. 하지만 에이전트가 매번 수동으로 "캡처해"라고 명령을 받아야 한다. 자율 에이전트라면 화면이 바뀌었을 때 스스로 반응해야 한다.

POC v2의 목표: 에이전트가 화면 변화를 감지하고, UI 요소를 자동으로 인식한다.

"dHash로 화면 변화를 잡아라"

OpenCV가 없으니 PIL + numpy만 써야 한다. 다행히 dHash (difference Hash)는 그 정도로 충분하다.

원리

  1. 이미지를 9×8로 축소
  2. 인접 픽셀끼리 비교 → 64비트 해시 생성
  3. 이전 해시와 해밍 거리 비교
  4. 거리가 임계값 미만이면 "변화 없음", 이상이면 "변화 감지"
def screen_changed(threshold=10):
    current_hash = dhash(capture_screen())
    dist = hamming_distance(prev_hash, current_hash)
    return dist > threshold

성능: 5ms 미만. 사실상 무시할 수 있는 수준이다. 60fps로 화면을 모니터링해도 여유다.

그래서 src/ui_extractor.py를 만들었다. clickable 요소의 bounds 좌표를 자동 추출하는 파서.

잘 돌아간다. 일반 앱에서는.

대항해시대 온라인은 달랐다

타겟 게임인 대항해시대 온라인 (com.linegames.uwogl)는 PC 클라이언트를 태블릿에 원격 렌더링하는 방식이다. SurfaceView 하나에 모든 게임 화면이 그려진다.

uiautomator 결과: 노드 6개, clickable 요소 0개.

SurfaceView 단일 렌더링이라 uiautomator는 사실상 미동작.
이걸 어떻게 인식하나.

결론: uiautomator로는 대항해시대를 인식할 수 없다. 다른 방법이 필요했다.

"OpenCV 없이 템플릿 매칭?"

결국 전통적인 방법으로 돌아갔다. 템플릿 매칭. 미리 찍어둔 버튼/아이콘 이미지를 화면에서 찾는 방식.

문제는 OpenCV가 Termux에서 빌드 안 된다는 것. 하지만 NCC (Normalized Cross-Correlation) 원리 자체가 복잡하지 않다.

numpy + PIL로 NCC 구현

def template_match(screen, template):
    # 2D 배열로 변환 → 슬라이딩 윈도우로 NCC 계산
    # 가장 높은 상관계수 위치 반환
    result = np.zeros((sh - th + 1, sw - tw + 1))
    for y in range(result.shape[0]):
        for x in range(result.shape[1]):
            region = screen[y:y+th, x:x+tw]
            result[y, x] = ncc(region, template)
    return np.unravel_index(result.argmax(), result.shape)
검색 범위소요 시간
전체 검색 (2560×1600)~53ms
ROI 한정 검색~8ms

ROI (관심 영역)를 지정하면 8ms. 충분히 실시간에 가깝다.

"화면별로 요소를 관리하자"

게임은 여러 화면으로 구성된다. 로그인 화면, 메인 화면, 상점, 전투... 각 화면마다 클릭 가능한 요소가 다르다.

이걸 YAML로 관리하자고 해서 src/element_db.py를 만들었다.

# wiki/30-elements/login.yaml
screen: login
elements:
  - name: id_input
    template: id_input.png
    roi: [100, 300, 500, 400]
  - name: login_button
    template: login_btn.png
    roi: [200, 600, 400, 700]

화면이 바뀌면 → 현재 화면 판별 → 해당 화면의 요소 DB 로드 → 템플릿 매칭으로 위치 탐색 → 자동 클릭. 이 흐름이다.

"버그 두 개 잡음"

POC v2를 만들면서 버그를 두 개 잡았다.

버그원인수정
요소 인식 실패키명 불일치 (actionable_elements vs elements)키명 일치 수정
/stop 명령 무시command_queue.get()에 5분 타임아웃5초마다 stop_event 체크로 변경

두 번째 버그가 좀 치명적이었다. 에이전트를 멈추고 싶은데 /stop을 보내도 5분을 기다려야 한다. 이건 실사용에서는 못 쓴다. stop_event를 5초마다 폴링하도록 바꿨다.

배운 것

  • SurfaceView 단일 렌더링 앱은 uiautomator가 안 통한다 — 템플릿 매칭으로 우회
  • OpenCV 없이도 numpy + PIL로 NCC 템플릿 매칭이 가능하다 (ROI 한정 시 8ms)
  • dHash로 화면 변화 감지는 5ms 미만으로 충분히 가볍다
  • 긴 타임아웃이 있는 큐에서 중지 명령을 처리하려면 event 폴링 패턴을 써야 한다
  • YAML로 화면별 요소를 정의하면 게임 화면 전환 관리가 체계적으로 된다
