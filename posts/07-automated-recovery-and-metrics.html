2026-04-23 Codex 연동 메트릭 목표 관리

문제: 고정관념에 갇힌 봇

런타임이 map-view라고 확신하고 항구 아이콘을 찾지 못해 무한 루프에 빠졌다. 분류기가 틀렸다는 가능성을 전혀 고려하지 않았다. 같은 액션을 15회 반복한 뒤 런타임이 스스로 중단했다.

회장님이 핵심을 짚었다: "연속 실패하면 현재 상황 분류가 없던 거라고 생각해라"

교훈 1: 내 판단을 검증하지 마, 다른 눈이 필요하다

X버튼 좌표 (653, 63)이 Codex 분석 결과라 무비판적으로 수용했다. 실제로는 픽셀 스캔 결과 (686, 12)이 올바른 위치였다. LLM 분석 결과를 검증 없이 좌표로 사용하면 안 된다.

원본 해상도에서 픽셀 스캔으로 밝은 클러스터를 찾고, 그 결과를 Codex 결과와 비교하는 과정이 필수다.

Codex 진단 루프

연속 5회 실패 → 루프 일시정지 → Codex에게 현재 스크린샷 전달:

Codex에게: "봇이 {state}라고 가정하고 {action}을 시도했으나 5+회 실패.
분류가 틀렸다. 실제 상태와 탭할 위치를 JSON으로 응답해라."
→ {"actual_state":"...", "tap_x":686, "tap_y":12}
→ 해당 좌표로 탭 → 화면 변화 확인 → 성공 시 카운터 리셋

10회 실패 시 Wiki 자가 개선도 실행: decision tables와 recognition map을 Codex가 직접 수정하고 텔레그램으로 diff를 보고한다.

고치면서 발견한 버그들

  • precondition 실패 → 성공 처리: skip-dialog가 precondition을 통과 못 했는데 "팝업 없음 → 성공"으로 기록. 이후 "직전 성공 액션 건너뜀"이 트리거되어 아무것도 안 하는 무한 루프.
  • 건너뜀 로직: 실제 화면 변화가 확인된 경우만 건너뛰도록 수정. 아니면 consecutive_fails를 증가시켜 Codex 진단으로 이어지게.
  • Codex git 체크: --skip-git-repo-check 누락으로 Codex가 실행조차 안 되는 문제.
  • /tmp 권한: Termux에서 /tmp에 쓰기 불가 → state_dir 내부로 변경.

장기/단기 목표 관리

봇이 "왜" 돌아가는지를 알아야 의미 있는 행동을 선택할 수 있다:

goals.json:
{
  "long_term": [
    {"id":"lt1", "description":"리스본~세비야 교역 10회", "progress":0, "target":10}
  ],
  "short_term": [
    {"id":"st1", "description":"출항 오피스 진입",
     "steps":["항구 메인 확보","출항 오피스 탭"], "current_step":0}
  ]
}

텔레그램 /goals 명령어로 진행도 조회. 주기적 상태 보고에도 목표가 포함된다.

인식/판단/제어 3단계 메트릭

파이프라인을 3단계로 분리하여 어디서 병목이 발생하는지 측정:

단계질문지표
인식"이 화면이 뭔가?"분류 안정성, 분류 속도
판단"무엇을 해야 하나?"precondition 실패율, 검증 통과율
제어"탭이 제대로 먹혔나?"화면 변화율, 목표 달성율

매 루프마다 loop_metrics 테이블에 기록. /metrics 명령어로 최근 50루프 집계 조회.

샘플 수집 파이프라인

상태별로 최대 20개 스크린샷 자동 저장 (MD5 중복 제거). 새 상태 감지 시 분석 스크립트로 클릭 영역 추출, 텔레그램으로 결과 이미지 전송.

핵심

자동화 봇이 막혔을 때, "액션을 바꿔라"가 아니라 "내가 보는 세상이 틀릴 수 있다"고 가정하는 것이 진짜 자가 복구다. 분류 신뢰 → 액션 선택 → 결과 검증 → 신뢰도 조정. 이 순환이 자가 학습의 핵심이다.
