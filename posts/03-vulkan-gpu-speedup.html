"106초가 3초가 됐다"
Adreno 840 Vulkan GPU 가속

2026-04-13 · 젬대버 프로젝트 · 읽기 7분

전 편에서 POC v1이 돌아가는 걸 확인했다. 하지만 비전 분석 한 번에 106초. 에이전트가 화면을 보고 반응하는 데 2분 가까이 걸린다. 이걸로 게임을 자동화할 수는 없다.

해결책은 하나. GPU 가속.

내 Y700에는 Adreno 840 GPU가 있고, Vulkan 1.4.335을 지원한다. llama.cpp에 Vulkan 백엔드를 붙이면 5~10배는 빨라질 거라고 예상했다.

예상: 5~10배
실제: 33배.

"사전 체크리스트"

Vulkan 빌드를 시작하기 전에 확인한 것들:

항목상태
Vulkan-Headers✅ 클론 완료
glslang (shader compiler)✅ Termux 패키지 설치
glslc 심볼릭링크✅ $PREFIX/bin/glslc → glslangValidator
vulkan-loader✅ mesa-vulkan-icd-freedreno 26.0.4
vulkan-tools✅ 1.4.349
GPU 확인✅ Adreno (TM) 840, Vulkan 1.4.335
spawn.h 스텁✅ 전체 스텁 함수 포함 완료

사전 조건은 다 맞춰져 있었다. 이전에 CPU 전용 빌드에서 겪은 spawn.h 문제도 이미 해결했고.

"shader 수백 개가 컴파일됐다"

cmake에 GGML_VULKAN=ON을 주고 빌드를 시작했다.

cd ~/phone-agent/llama.cpp
mkdir build-arm64-vk && cd build-arm64-vk
cmake .. -DGGML_VULKAN=ON \
  -DCMAKE_BUILD_TYPE=Release \
  -DLLAMA_CURL=OFF

Vulkan shader 컴파일이 시작됐다. .spv 파일이 수백 개 생성됐다. 각 shader가 Vulkan SPIR-V 바이너리로 컴파일되는 거다.

libggml-base.so, libggml-cpu.so는 순조롭게 빌드됐다. 하지만 llama-server 빌드는 1차 시도에서 완료하지 못했다. 시간 초과로 중단된 것 같다.

"2차 빌드에서 결국 성공"

기존 build-arm64-vk 디렉토리에서 cmake --build를 재개했다.

cmake --build . -j4

warning이 수십 개 떴다. jinja 관련 noreturn 경고, 타입 캐스팅 경고... 하지만 에러는 없었다.

✅ llama-server 빌드 완료

최종 바이너리: 14.1MB. libggml-vulkan.so.0 링크 확인. llama.cpp v8 (commit 227ed28).

이게 이 프로젝트에서 가장 뿌듯한 순간 중 하나였다. 태블릿에서 Vulkan GPU 백엔드가 달린 LLM 서버를 직접 빌드한 거다.

"GPU가 잡혔다"

서버를 실행했다:

./bin/llama-server \
  -m ~/phone-agent/models/google_gemma-4-E4B-it-Q4_K_M.gguf \
  --mmproj ~/phone-agent/models/mmproj-F16.gguf \
  -ngl 99 -c 2048 -b 32 --threads 4 --port 8080

로그가 떴다:

LLM server listening at http://0.0.0.0:8080

GPU: Vulkan0 (Adreno 840) 백엔드 사용
KV cache: 112 MiB GPU 할당
mmproj (vision+audio encoder): 944 MiB, Vulkan0 백엔드
Flash Attention: 자동 활성화
4 슬롯 초기화, prompt cache 활성화

944 MiB의 비전 인코더가 GPU에 올라갔다. KV cache도 GPU에 할당됐다. Flash Attention도 켜졌다. 모델 전체가 GPU에서 돌아가고 있다.

"106초 → 3.2초"

속도 테스트 결과:

항목CPU 전용Vulkan GPU개선배율
Prompt 속도~1,000+ ms/tok96.7 ms/tok (10.3 tok/s)~10x
Generation 속도~1,000+ ms/tok108.8 ms/tok (9.2 tok/s)~10x
총 응답시간~106초3.2초~33x

토큰별 속도는 10배 빨라졌는데, 총 응답시간은 33배 빨라졌다. 이건 Flash Attention과 KV cache 최적화가 복합적으로 작용한 결과다.

📊 왜 10배인데 33배?

토큰 속도만이 아니다. GPU에 올라간 모델은 프롬프트 처리, 메모리 전송, 캐시 히트 등 전체 파이프라인이 가속된다. 특히 비전 인코더 (944MB)가 GPU에서 돌기 때문에 이미지 처리 부분의 이득이 크다.

3.2초면 실시간 게임 자동화에 충분하다. 화면이 바뀌고 → 3초 안에 AI가 분석하고 → 다음 동작을 결정한다.

"삽질 정리"

Vulkan 빌드 과정에서 겪은 문제들:

1. spawn.h (CPU 빌드 때도)

Android Bionic에 POSIX spawn이 없다. 전체 스텁 헤더를 작성해야 했고, 최소 스텁으로는 llama-server가 빌드 안 됐다.

2. glslc 링크

Termux에는 glslangValidator만 있고 glslc가 없다. 심볼릭링크로 연결해야 llama.cpp의 cmake가 shader를 컴파일할 수 있다.

ln -s $PREFIX/bin/glslangValidator $PREFIX/bin/glslc

3. 빌드 중단

1차 시도에서 시간 초과/중단으로 llama-server 빌드가 미완료됐다. 하지만 중간产物가 남아 있어서 2차 시도에서 cmake --build로 이어서 빌드할 수 있었다.

4. -ngl 99

-ngl 99는 모델 레이어를 최대한 GPU에 올리라는 의미다. VRAM이 허용하는 한 모든 걸 GPU에 올려서 추론 속도를 극대화한다. Y700의 14GB RAM 덕분에 944MB 비전 인코더 + KV cache를 전부 GPU에 올릴 수 있었다.

배운 것

  • llama.cpp의 Vulkan 백엔드는 Android에서도 잘 돌아간다 (Adreno 840 확인)
  • GPU 가속은 토큰 속도뿐 아니라 전체 파이프라인을 가속한다 (10x → 33x)
  • Flash Attention은 자동 감지/활성화된다 — 별도 설정 불필요
  • 빌드가 중단돼도 cmake incremental build로 이어서 할 수 있다
  • -ngl 99로 전 레이어 GPU 할당 시 VRAM 사용량을 미리 확인해야 한다
  • vulkaninfo로 GPU/Vulkan 버전을 확인하고, 호환성을 미리 체크하면 시간 낭비를 줄일 수 있다
