"무시" 경로의 함정
LLM 자동화에서 배운 6가지 문제해결 패턴

2026-04-24 설계 패턴 자가 개선 메모리 관리

문제: 732번 루프에서 아무것도 안 함

런타임이 정상 작동 중인 것처럼 보였다. 매 루프마다 캡처하고, 분류하고, 로그를 남겼다. 하지만 실제로는 같은 화면에서 732번째 "화면 변화 없음 — 스킵"을 출력하고 있었다.

Codex 진단은 연속 5회 실패 시 발동하도록 설정되어 있었는데, 스킵은 성공도 실패도 아니었기 때문에 카운터가 절대 도달하지 않았다. 감독관은 정확히 진단했지만 코드를 수정할 권한이 없어서 보고만 남기고 끝났다.

하루 종일 수정한 코드를 분석해보니, 6가지 구조적 패턴이 드러났다.

패턴 1: "성공"의 정의를 다시 생각하기

dismiss-generic 액션이 "화면 변화 있음"을 검증 기준으로 삼았다. 팝업을 닫으면 화면이 바뀌니까 성공. 하지만 닫힌 자리에 새 팝업이 뜨면? 화면은 변했고, 검증은 통과했고, 실패 카운터는 리셋됐다. 실제로는 같은 상태에 갇혀 있었다.

// Before: "뭔가 바뀌었는가?"
if (screen_changed) success()

// After: "목표 상태로 나아갔는가?"
if (screen_changed AND state_changed) success()
else if (screen_changed AND same_state) fail_count++

검증 기준은 수단이 아니라 목표를 봐야 한다. "화면이 바뀌었다" ≠ "문제가 해결됐다".

패턴 2: "무시" 경로에 실패 누적하기

이번 사건의 핵심이다. "화면 변화 없음 → 스킵" 경로는 시스템의 공백이었다. 액션도 아니고, 성공도 아니고, 실패도 아니었다. 카운터가 누적되지 않으니 Codex도 불리지 않고, 루프는 732번까지 멈추지 않았다.

// Before: 무시 (공백)
if not screen_changed:
    log("skip")
    continue

// After: 무시도 실패로 간주
if not screen_changed:
    consecutive_fails++
    log(f"skip (fails={consecutive_fails})")
    if consecutive_fails % 10 == 0:
        force_back()  // 정체 상태 해제
    continue

시스템에 "무시" 경로가 있으면 거기에도 실패 카운트를 달아라. 무시가 쌓이면 정체 상태다. 정체는 실패보다 위험하다 — 실패는 알아챌 수 있지만, 정체는 시스템이 "정상 작동 중"이라고 믿게 만든다.

패턴 3: 하위 시스템 설정을 외부 파일로 분리

OCR 키워드 분류기의 상태 정의가 코드 안에 하드코딩되어 있었다. "선단 관리" 화면이 event로 분류되는 버그를 고치려면 코드를 수정해야 했는데, Codex는 코드 수정 권한이 없었다.

해결: ocr_classifier.py의 STATES 딕셔너리를 config/ocr_states.json으로 분리했다. 매 호출마다 파일을 새로 읽는다.

# ocr_classifier.py
def _load_states() -> dict:
    with open("config/ocr_states.json") as f:
        return json.load(f)  # 매번 새로!

def classify_state(text):
    STATES = _load_states()  # Codex가 수정하면 즉시 반영
    ...

코드 수정은 1회. 이후 Codex가 JSON을 직접 편집하면 다음 루프부터 즉시 반영된다. 핫픽스 가능한 설정 계층.

패턴 4: 무거운 작업 전 리소스 3단계 확인

Codex 진단 실행 중 Termux가 통째로 죽었다. Android OOM killer가 프로세스를 강제 종료한 것. 원인: 게임 앱 + 젬대버 + Codex가 동시에 12GB/15GB를 점유.

파이프라인에 무거운 단계가 있으면 실행 전 3단계를 밟아라:

  1. 확보 — _free_memory(): gc.collect() + 오래된 스크린샷 삭제 (992MB → 300개로 제한)
  2. 재확인 — _has_memory_for_codex(): /proc/meminfo에서 MemAvailable 확인 (최소 500MB)
  3. 대기 — 여전히 부족하면 텔레그램 알림 + 1분 대기 후 재시도
free_memory()
if not has_memory_for_codex():
    alert("메모리 부족, 루프 일시정지")
    sleep(60)
    continue
codex_diagnose()  # 안전하게 실행

패턴 5: 자가 개선 대상을 두 계층으로 분리

Codex 감독관이 무엇을 수정할 수 있는가? 모든 것을 열어주면 위험하고, 아무것도 못 열어주면 쓸모가 없다.

계층대상반영 방식위험도
데이터wiki, goals.json, ocr_states.json즉시 (다음 루프)낮음
코드runtime.py, ocr_classifier.py승인 후 재시작높음

데이터 계층은 Codex가 자유롭게 수정. 코드 계층은 approval_requested: true로 표시하고 회장님 승인 후 적용. 이 분리가 없으면 감독관이 진단만 하고 끝나는 "관측자"에 머문다.

패턴 6: 행위를 시각적으로 기록하면 사후 분석이 정확해진다

오버레이 APK로 탭 위치에 손가락 커서를 표시한다. 단, 인식용 캡처에서는 끄고 제어용 캡처에서만 켠다:

// 인식 캡처 (분류용) — 커서 끄기
overlay.send("hide_cursor")
screenshot = capture()

// 제어 캡처 (검증용) — 커서 켜기
overlay.tap(x, y)  // 커서 표시
control.tap(x, y)
verify_screenshot = capture()

이렇게 하면 스크린샷 자체가 "어디를 탭했고 결과가 어떻게 되었는지"를 기록한다. Codex가 사후 분석할 때 스크린샷만 봐도 탭 위치와 결과를 알 수 있다. 시각적 컨텍스트 없이 좌표 숫자만 보는 것보다 정확도가 크게 올라간다.

요약

패턴한 줄

설정 외부화JSON + 매번 재로딩 = LLM 핫픽스

핵심

자동화 시스템에서 가장 위험한 상태는 "고장"이 아니다. "정상 작동 중"이라고 믿게 만드는 정체다. "무시" 경로에 카운터를 달고, 성공의 정의를 목표에 맞추고, 설정을 외부화해서 자가 개선 경로를 열어두는 것. 이 세 가지가 정체 상태를 방지하는 핵심이다.
