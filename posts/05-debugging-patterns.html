게임 자동화에서 배운 7가지 문제해결 패턴

2026-04-19 · 젬대버 프로젝트 · 읽기 12분

대항해시대 오리진이라는 모바일 게임의 항해 루프를 자동화하는 프로젝트를 진행하고 있다. ADB로 태블릿을 제어하고, 템플릿 매칭 + 규칙 기반 분류기로 화면 상태를 인식한 뒤, LLM이 액션을 선택하는 구조다.

3일간의 개발에서 예상치 못한 버그들이 연달아 발생했고, 각각을 해결하면서 반복되는 패턴이 보이기 시작했다. 이 글은 그 패턴들을 정리한 것이다.

게임 자동화뿐만 아니라, 외부 시스템과 상호작용하는 모든 자동화 프로젝트에 적용할 수 있는 원칙들이다.

패턴 1: 사이드이펙트 숨겨진 버그

문제

검사 함수가 검사와 상태 변경을 동시에 수행하면, 호출 순서에 따라 결과가 달라진다.

사례

def screen_changed(self, screenshot_path):
    current_hash = capture_hash(screenshot_path)
    changed = hamming(current_hash, self._last_hash) > threshold
    self._last_hash = current_hash  # ← 상태 변경!
    return changed

메인 루프에서 screen_changed()를 호출하면 _last_hash가 업데이트된다. 그런데 액션 실행 후 verify_result()에서 같은 스크린샷으로 다시 screen_changed()를 호출하면, 이미 해시가 같으므로 항상 "변화 없음"을 반환한다.

해결

# 순수 함수로 분리
def capture_hash(screenshot_path):
    return dhash(load_image(screenshot_path))

# 상태 변경은 별도 메서드
def update_hash(self, h):
    self._last_hash = h
💡 원칙

검사 함수는 상태를 변경하지 않는다. 순수 함수로 분리하면 버그 원인을 추적하기 훨씬 쉬워진다.

패턴 2: 거짓 양성 캐스케이드

문제

하나의 템플릿이 여러 화면에서 매칭되면, 전체 상태 인식이 무너진다.

사례

템플릿의도실제원인
popup_close_x.png팝업 닫기모든 화면에서 conf=1.0X 모양이 너무 단순
wh_back_arrow.png창고 뒤로가기미니맵에서도 0.76~1.0버튼 형상 유사
dialog_auto_btn.png대화 자동진행창고에서도 1.0버튼 형상 유사

하나의 거짓 양성이 상태 분류를 틀리게 만들고, 잘못된 상태에서 잘못된 액션이 선택되는 연쇄 반응이 일어난다.

해결 (강도 순)

  1. 삭제 — 단순 형상 템플릿은 아예 제거
  2. ROI 제한 — 탐색 영역 축소
  3. 우선순위 — 검색 순서 지정
  4. 규칙 기반 대체 — 색상/밝도 기준으로 판별

가장 효과적이었던 것은 규칙 기반 대체였다:

# 템플릿 대신 중앙 밝도 기준
center_brightness = frame[100:350, 100:620].mean()
is_popup = center_brightness > 120  # 항구=38, 팝업=182
💡 원칙

단순 형상(버튼, 화살표, X)은 템플릿 매칭에 부적합하다. 규칙/색상/레이아웃 기반으로 대체하라.

패턴 3: precondition-실행-verification 타이밍 불일치

문제

액션은 성공했는데 verification이 다음 프레임을 보거나, 아직 UI가 반영되지 않은 상태를 검사한다.

사례

미니맵 버튼 탭 → 화면 변화 있음(hamming=72) → verification map_opened 실패. 미니맵이 열리는 데 시간이 걸리거나, NPC 대화창이 겹쳐서 분류가 혼동된다.

해결

verification을 강제하지 않고 "실패해도 다음 루프에서 자연 복구"로:

verification:
  - id: map_opened
    method: rule
    rule: "state_is map-view"
    on_fail: skip  # 실패해도 다음 루프에서 상태 분류로 확인
💡 원칙

비동기 UI에서는 verification 타이밍을 강제하지 말라. "실패해도 다음 루프에서 자연 복구" 패턴을 사용하라.

패턴 4: 액션 선택 무한 루프

문제

A precondition 실패 → B 선택 → B precondition 실패 → A 선택 → 영원 반복.

사례

dismiss-generic(score=200)과 skip-dialog(score=190)가 교차 실행. 두 액션 모두 precondition이 없으면 성공 처리되어 번갈아 선택되고, open-minimap(score=100)은 절대 선택되지 않는다.

해결

# ❌ 개별 건너뛰기 (교차 실행 발생)
if last_action == goal["action"] and last_success:
    skip(goal)

# ✅ 계열 단위 건너뜀
SKIP_FAMILIES = {"dismiss-generic", "handle-login-popup", "skip-dialog"}
if last_action in SKIP_FAMILIES and last_success:
    for goal in goals:
        if goal["action"] in SKIP_FAMILIES:
            skip(goal)
💡 원칙

"성공 처리되는 noop 액션"의 건너뜀 기준은 액션 단위가 아니라 계열 단위로.

패턴 5: 게임 UI 중첩으로 인한 좌표 불일치

문제

화면 상태(메인/시설/대화창)에 따라 같은 액션이 다른 곳을 탭한다.

사례

화면 상태미니맵 버튼결과
항구 메인존재탭 성공 ✅
교역소 내부없음다른 UI 건드림 ❌
NPC 대화창가려짐대화창으로 감 ❌

해결

"목표 액션을 실행할 수 있는 상태인가?"를 먼저 검사하고, 아니면 복구 액션을 선행:

대화창 감지 → skip-dialog → 메인 복귀 확인 → open-minimap
💡 원칙

게임 자동화에서는 "현재 화면이 목표 액션을 실행할 수 있는 상태인가?"를 항상 검증하라.

패턴 6: 환경 의존성 (ADB/화면/게임)

문제

코드는 맞는데 실행 환경에서 실패한다.

사례

환경 문제증상해결
ADB 터치 불응hamming=0재시도 로직
화면 꺼짐26KB 검은 화면_ensure_screen_on()
게임 보안정책 종료프로세스 사라짐watchdog 자동 재시작
/tmp 접근 불가PermissionErrorworkspace 내 임시파일
Claude Code proot 제한glibc 바이너리 실행 불가proot-distro Ubuntu
💡 원칙

모든 외부 I/O는 실패 가능성을 가정하고 재시도/복구 레이어를 두라.

패턴 7: 좌표 하드코딩 vs 동적 탐지

문제

고정 좌표는 화면 레이아웃 변화에 취약하다. 그리고 "좌표가 틀렸다"는 가정에 너무 빨리 앵커링하기 쉽다.

사례

미니맵 버튼 좌표를 찾기 위해 2시간 30분을 소모:

  1. 250px 간격 그리드 스캔 → HIT → 재현 안 됨
  2. 정밀 스캔 → 다른 HIT → 또 재현 안 됨
  3. Claude Code로 현재 화면 분석 → 교역소 내부였음 (미니맵 버튼 자체가 없음)

원래 좌표 (2828,384)가 맞았다. 문제는 좌표가 아니라 화면 상태였다.

해결

탐지 우선순위:
1. 템플릿 매칭 (동적)
2. LLM 좌표 추론 (준동적)
3. Fallback 하드코딩 (최후 수단)
💡 원칙

동적 탐지를 1순위로. 하드코딩은 최후 수단으로. 그리고 "좌표가 틀렸다"고 가정하기 전에 현재 화면 상태를 먼저 확인하라.

문제해결 프로세스: 진단 선행

이 패턴들에서 도출한 개선된 문제해결 프로세스:

실패 발생
  ↓
① 현재 화면 상태 파악 (10초)
  ↓
② "목표 액션이 가능한 상태인가?"
  ├─ 아니오 → 상태 복구 먼저
  └─ 예 → 원인 좁히기
       ↓
③ 로그에서 수치 분석 (hamming, confidence, score)
  ↓
④ 최소 수정 → 1회 테스트 → 성공이면 커밋

체크리스트

  • ☐ 현재 화면을 먼저 분석했는가?
  • ☐ 목표 액션이 가능한 상태인가?
  • ☐ 로그에서 수치를 확인했는가?
  • ☐ 수정 전에 가설을 적었는가?
  • ☐ 1회 테스트로 검증했는가?
🎯 핵심

"어떻게 고칠까"보다 "왜 실패했을까"를 먼저.

미니맵 좌표 문제는, 현재 화면을 먼저 분석했다면 2분에 끝났을 문제를 2시간 30분이나 썼다. "좌표가 틀렸다"는 첫 번째 가설에 너무 빨리 앵커링한 것이 원인이었다.

이 패턴들은 게임 자동화뿐만 아니라 RPA, 크롤링, IoT 제어 등 외부 시스템과 상호작용하는 모든 자동화에 공통적으로 적용된다. 특히 "진단 선행" 원칙은 가장 비용이 싸면서 가장 큰 효과가 있는 개선이었다.
