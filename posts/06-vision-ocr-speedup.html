"471초에서 4초까지"
Gemma 4 비전 OCR 속도 최적화

2026-04-21 읽기 6분

게임 자동화 봇 젬대버에서 화면 텍스트를 읽기 위해 Gemma 4 비전 모델을 도입했다. 초기 결과는 참담했다 — OCR 한 번에 7분 51초. 이를 어떻게 4초로 줄였는지 기록한다.

환경

  • 기기: Lenovo Tab Y700 (Adreno 840, Vulkan 1.4)
  • 런타임: Termux + llama.cpp (Vulkan GPU)
  • 모델: Gemma 4 E4B Q4_K_M (5.1GB) / E2B Q4_K_M (3.46GB)
  • 이미지: 게임 스크린샷 3040×1904 (원본)
  • 용도: 게임 UI 텍스트 OCR (한국어)

1단계: 기준선 측정 — 왜 이렇게 느린가

처음 E4B로 원본 이미지를 보냈더니 471초 (0.6 tok/s). GPU 스로틀링인지, 이미지가 무거운 건지 원인을 몰랐다.

streaming으로 TTFT 분리

응답을 스트리밍으로 받아 첫 토큰 시간(TTFT)과 생성 시간을 분리했다:

TTFT (이미지 인코딩): 0.3초  ✅ 매우 빠름
Generation (토큰 생성): 19.3초 ❌ 여기가 병목

이미지 처리는 0.3초면 끝난다. 문제는 모델이 불필요하게 긴 설명을 생성하는 것. "화면 속 모든 text"라는 프롬프트에 대해 텍스트 외에 분석, 추론, 설명까지 256토큰을 생성하고 있었다.

2단계: E4B vs E2B 모델 비교

E4B(5.1GB)가 무거워서 E2B(3.46GB)로 교체. GPU 메모리 사용량:

E4B: ~8.5GB (GPU 메모리 부족으로 간헐적 크래시)
E2B: 1,285 MiB (여유룸)

속도는 비슷했지만 인식률에서 차이가 있었다:

E4BE2B
속도243~471초173~236초
인식률"위대한 항로의 출석" ✅"성장기의 길드 1단계" ❌

E2B가 빠르지만 정확도가 떨어졌다. 하지만 최적화 후 결과를 보면 의외로 인식률이 개선되었다.

3단계: Claude Code에게 조언 구하기

원인 분석으로만 시간을 낭리지 않기 위해 Claude Code에 상황을 설명하고 최적화 방법을 물었다. 핵심 조언:

  • -ngl 99 — 모든 레이어 GPU 오프로드
  • -fa on — FlashAttention 활성화
  • -ub 512 — 물리적 배치 사이즈 증가
  • -ctk q8_0 -ctv q8_0 — KV 캐시 8비트 양자화
  • --cache-reuse 256 — 프롬프트 캐시 재사용
  • 이미지를 896×896으로 미리 줄이면 5~10배 빨라진다

4단계: 이미지 크기별 인식률 비교

최적화된 서버에서 5가지 크기로 테스트:

크기속도t/s핵심 텍스트 인식
720×45025초9.46/8 ⭐
896×56019초13.42/8
1280×80027초9.72/8
1920×120025초9.52/8
원본 3040×190422초13.34/8

의외로 720×450이 인식률이 가장 좋았다. 큰 이미지가 오히려 노이즈를 유발하는 것 같다.

5단계: 프롬프트 최적화 — 결정적 한 방

TTFT 분석에서 병목이 이미지 처리가 아닌 토큰 생성이라는 걸 알았다. 프롬프트를 바꿨다:

# Before
"화면 속 모든 text"

# After
"이미지의 텍스트만 줄바꿈 유지해 추출. 설명/추론 금지."
max_tokens: 128 (이전 1024)
BeforeAfter
속도25초4초
gen tokens25645
인식률6/86/8

25초 → 4초. 이미지 크기도, 서버 옵션도 바꾸지 않았다. 프롬프트만 바꿨다.

최종 서버 옵션

llama-server \
  -m google_gemma-4-E2B-it-Q4_K_M.gguf \
  --mmproj mmproj-gemma-4-E2B-it-f16.gguf \
  -ngl 99 -fa on -c 2048 -b 2048 -ub 512 \
  -ctk q8_0 -ctv q8_0 -t 6 \
  --cache-reuse 256 -np 1

요약

단계속도비고
E4B 원본, 미최적화471초0.6 tok/s
E2B + 서버 최적화173초1.4 tok/s
E2B + 720×45025초9.4 tok/s
E2B + 프롬프트 최적화4초11.2 tok/s

배운 것

  1. 병목 분석 먼저. 스트리밍으로 TTFT와 generation 시간을 분리하면 병목이 바로 보인다. "이미지가 무거워서 느리다"는 가설이 틀렸다.
  2. 프롬프트가 속도를 결정한다. "모든 텍스트를 나열해줘"는 256토큰 설명을 유도한다. "추출만 해. 설명 금지"로 45토큰으로 줄였다.
  3. 큰 이미지가 항상 좋은 건 아니다. 720×450이 원본(3040×1904)보다 인식률이 좋았다.
  4. E2B가 E4B보다 실용적이다. 1/3 크기, GPU 메모리 1/6, 속도 비슷. 인식률은 프롬프트로 보정 가능.
  5. Claude에게 물어봐라. 이미 해결한 사람에게 물어보는 게 가장 빠르다.

다음 단계: 이 OCR 파이프라인을 젬대버 런타임에 통합. 4초면 게임 자동화 루프에 충분히 실시간으로 사용할 수 있다.
