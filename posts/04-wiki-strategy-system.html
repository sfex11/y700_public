"64개 액션부터 전략 시스템까지"
게임 자동제어 지식베이스 완성

2026-04-15 · 젬대버 프로젝트 · 읽기 8분

에이전트가 게임을 자동으로 하려면 세 가지를 알아야 한다. "지금 어디에 있는가", "무엇을 할 수 있는가", "지금 뭘 해야 하는가". 앞의 두 개는 wiki에 정리해뒀는데, 마지막 "뭘 해야 하는가"가 빠져 있었다.

이번 작업에서 42개 액션을 64개로 확장하고, 전략 판단 레이어 전체를 설계·구축했다.

42 → 64개 액션: 누락된 시스템을 채우다

기존 42개 액션은 항해·교역·전투·복구 위주였다. 게임의 다른 시스템들이 빠져 있었다.

"전략을 누가 판단하나" — 아키텍처 논쟁

64개 액션이 있어도 "지금 in-port인데 교역을 할지 수리를 할지"는 상황마다 다르다. 이 결정을 누가, 어떻게 하느냐가 핵심 문제다.

세 가지 방식을 비교했다

방식장점단점
A. 하드코딩 룰빠르고 확실상황 변화에 둔감
B. LLM이 전략 문서 읽고 판단유연초소형 모델은 흔들림
C. 전략은 컴파일, LLM은 선택만안정+유연전략 수정 시 재컴파일

결론: C를 기본으로 하되 "C + 얇은 룰 엔진 + 아주 작은 LLM 판정"으로 가는 게 가장 안전하다. gemma 4에게 "전략을 생각하게" 하면 흔들리고, 대신 전략은 미리 구조화하고 LLM은 분류·선택·예외 확인만 맡겼다.

6개 전략 모드 FSM

게임 상태(항해/항구/전투)와 별개로, 봇의 전략 목표를 관리하는 별도 상태기계를 만들었다.

BOOTSTRAP → MONEY_MAKING ↔ RECOVERY
                  ↕
                GROWTH
                  ↓
           COMBAT_ESCAPE / COMBAT_ACTIVE
모드목표전환 조건
BOOTSTRAP앱 실행→로그인→기본루프부팅 완료 시
MONEY_MAKING교역으로 수익 극대화기본 모드
RECOVERY내구도/보급/자금 복구내구도<40% 또는 골드<3000
GROWTH선박/항해사/장비 업그레이드골드>100K + 내구도>70%
COMBAT_ESCAPE전투 회피/도주적 조우 + 전투력 열위
COMBAT_ACTIVE전투로 이득 획득적 조우 + 전투력 우위

같은 in-port 상태에서도 전략 모드가 RECOVERY면 수리/보급 후보만 나오고, MONEY_MAKING이면 교역/시장 후보가 나온다. "같은 상태인데 왜 행동이 다르지?"를 가장 깔끔하게 해결하는 구조다.

Goal Arbitration — 점수형 우선순위

단순 if/elif 대신 각 goal에 점수를 매기고 최고 점수의 액션을 실행하는 방식을 썼다.

state: in-port
mode: MONEY_MAKING
goals:
  sell_high:  cargo_has_goods AND 시세≥110% → 80점
  buy_cheap:  cargo<90% AND gold>5000 AND 시세≤90% → 75점
  check_market: 시세_미확인 → 70점
  invest:     gold>50000 AND 쿨타임완료 → 60점

전략 수정이 테이블만 바꾸면 되니까 유지보수가 쉽다. 코드 수정 없이 문서만 고치면 된다.

gemma가 하는 일 (3가지뿐)

초소형 로컬 모델에게 맡기는 건 3가지다.

1. 상태 분류 (~1.4초)

Look at the screenshot. Which state is this?
Options: sailing, in-port, trade, combat, event, recovery
Answer only the state name.

2. 후보 선택 (~0.8초)

STATE=in-port
CANDIDATES: 1.check-prices 2.buy-goods 3.repair
Choose the best action. Answer only the number.

3. Yes/No 검증 (~0.8초)

Is this screen showing repair menu?
Answer only YES or NO.

"지금 돈이 부족하니 장기적으로 교역을 하고 내구도도 고려해서…" 같은 복잡한 추론은 gemma에게 맡기지 않는다. 전략은 이미 decision table에 컴파일되어 있으니까.

전체 wiki 구조 (108개 파일)

wiki/
├── readme.md              ← 안내서
├── actions/       64개    ← 할 수 있는 것
├── states/         8개    ← 지금 어디인가
├── strategies/     7개    ← 어떤 전략을 쓸까
├── compiled/      20개    ← 런타임용
│   ├── decision_tables/   state×mode → goal 점수
│   ├── checklists/        state별 팩트 추출 항목
│   └── prompts/           gemma용 초단문 프롬프트
└── bot/            9개    ← 제어 로직

런타임 흐름

1사이클 4~10초, gemma 호출 1~2회.

  1. 스크린샷 캡처 (adb, 200ms)
  2. 상태 분류 (gemma, 1.4s)
  3. 팩트 추출 (OCR/템플릿 우선, 0.1~1s)
  4. 전략 모드 유지 또는 전환 (룰 엔진, <1ms)
  5. Decision Table → goal 점수 계산 (룰 엔진, <1ms)
  6. 최고 goal → 액션 후보 2~4개 선정
  7. gemma 후보 선택 (0.8s)
  8. 액션 실행 (ADB, 1~5s)
  9. 결과 확인 (템플릿/gemma)
  10. 실패 시 fallback, 성공 시 루프 계속

배운 것

  • 초소형 LLM에게 "전략을 이해시키는 것"보다 전략을 미리 컴파일해서 아주 작은 선택 문제만 주는 것이 훨씬 현실적이다
  • 전략 모드를 게임 상태와 분리하면 "같은 화면에서 다른 행동"을 깔끔하게 설명할 수 있다
  • Goal Arbitration(점수형)이 if/elif보다 유지보수에 유리하다 — 코드 수정 없이 문서만 고치면 된다
  • 전략 판단은 매 프레임이 아니라 상태 전이 시점에만 해야 지연과 정책 일관성이 모두 좋다
  • Claude Code에 충분한 프롬프트와 시간을 주면 22개 파일을 한 번에 품질 좋게 생성한다
